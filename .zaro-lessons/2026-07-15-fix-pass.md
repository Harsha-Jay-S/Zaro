# Zaro fix pass — root-cause lessons (2026-07-15)

Ten audit findings fixed. Most collapse into four root-cause patterns worth
remembering for the next iteration.

## Pattern 1 — Code-as-data (findings #2)
**Root cause:** model/agent output was interpolated into `python3 -c "...'$summary'..."`
source. Any quote/apostrophe/`$(...)` in a review summary became executable text;
under `set -e` a syntax error killed the whole daemon.
**Fix:** pass every model-derived value via `argv` into a **quoted** heredoc
(`python3 - "$summary" <<'PY'`), and wrap the state-write call in `|| log warning`
so a bad write can never take the daemon down.
**Lesson:** never build an interpreter's *source* from untrusted output. Pass data
as data (argv/env/stdin). This is a class, not a one-off — audit every `python3 -c`
and `sh -c` that embeds a variable.

## Pattern 2 — Host-wide vs scoped side effects (finding #3)
**Root cause:** cleanup used `pgrep -f "python3 client.py" | sudo kill` — a
host-wide, privileged sweep that would kill *any* matching process, including
other opencode sessions.
**Fix:** run each `opencode run` under `setsid` (its own process group) and reap
only that group (`kill -- -$pgid`). Track the group id in a file so stop/interrupt
handlers can reap an in-flight run. No `sudo`, no pattern matching.
**Lesson:** to clean up "my children," use the process group / tree, never a
global name pattern. Reparented orphans keep the group id — pattern-matching
can't tell them from a stranger's process.

## Pattern 3 — Wrong config layer (finding #1) — the expensive trap
**Root cause:** the RAG plugin looked "dead" because it wasn't in the plugin
array. But injection-log residue and a built embeddings file suggested it *had*
run — a contradiction that, unreconciled, would have sent the fix at the wrong
file. Reality: those were 07-12 test probes; the plugin genuinely wasn't loaded
in sessions; `~/.config/opencode/opencode.json` **is** the authoritative config
(verified: opencode logs `loading path=.../opencode.json` then `[zaro] Ready`).
**Lesson:** this is exactly the AGENTS.md debugging rule — "editing a low-priority
source is useless if a higher-priority one overrides it." Before fixing a
config-driven bug, *prove which config the runtime actually loads* (here: run it,
watch the boot log). And when evidence contradicts your premise (log says it ran,
config says it can't), stop and reconcile before editing.

## Pattern 4 — Doc/impl drift (findings #7, #8) & dead code (#9, #10)
**Root cause:** the agent prompt described a `special:"review"` trigger and a
report format the shell couldn't parse; zaro.js carried an always-true gate and a
vestigial guard from earlier designs.
**Fix:** make the doc describe the real mechanism (milestone counter), add the
`Summary:` line the shell actually greps, delete the dead branches.
**Lesson:** when two files must agree (agent report ↔ shell grep; curriculum
domains ↔ domain-map), a fix to one is incomplete until the other is checked.
The smoke test now encodes the domain-map ↔ curriculum invariant so it can't
silently drift again.

## Meta-lesson — pre-seed invariants instead of per-cycle patching
Finding #6 (missing domain keywords) existed because the agent was *supposed* to
add a domain-map entry on each study cycle — a per-cycle obligation that silently
failed. Pre-seeding all 13 domains up front removes the obligation entirely. Prefer
a stable invariant checked by a test over a runtime step that can be skipped.
